# ADR-0005: BullMQ over AWS SQS

**Status:** Accepted
**Date:** 2025-01-01

## Context

Image generation is async — jobs are queued and processed by a worker. We need a reliable job queue with:

- Automatic retries with exponential backoff
- Dead-letter queue for persistently failing jobs
- Job status tracking (for SSE updates)
- Local development support without AWS credentials

Options considered:

- **BullMQ** — Redis-backed, feature-rich, widely used with Node.js
- **AWS SQS** — managed, pay-per-use, requires AWS credentials
- **RabbitMQ** — AMQP, powerful routing, operational overhead
- **Inngest** — managed job queue service, newer player

## Decision

We will use **BullMQ with Redis** for the job queue.

Key factors:

1. **Railway-native:** Redis is already in our stack for JWT blacklisting and rate limiting — BullMQ reuses it with zero additional infrastructure
2. **No AWS credentials in dev:** SQS requires AWS credentials in local development; BullMQ works with Docker Compose Redis
3. **Job status in Redis:** BullMQ stores job state in Redis, which our `job:status:{jobId}` key pattern reads directly for SSE updates
4. **Feature-rich:** Delayed jobs, repeatable jobs (quota reset cron), dead-letter queues, job events — all built in
5. **Cost:** BullMQ is free; SQS costs scale with message volume

## Consequences

- ✅ Works in Docker Compose with no external credentials
- ✅ Job status available directly from Redis for SSE polling
- ✅ Built-in cron for monthly quota resets
- ⚠️ Redis is a single point of failure for both sessions and jobs — mitigated by Railway-managed Redis with persistence
- ⚠️ BullMQ queues are lost if Redis data is lost without AOF persistence (configure Railway Redis with persistence enabled)
