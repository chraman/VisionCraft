# ADR-0007: RS256 JWT with Refresh Token Rotation + Family Revocation

**Status:** Accepted
**Date:** 2025-01-01

## Context

Authentication requires secure, stateless tokens with revocation capability. Requirements:

- Short-lived access tokens (protect against interception)
- Persistent sessions (users should not log in every 15 minutes)
- Immediate revocation on logout or security event
- Detect token theft via refresh token reuse

Options considered:

- **HS256 JWT** — symmetric, simpler, but can't verify on multiple services without sharing the secret
- **RS256 JWT** — asymmetric, services verify with public key (no secret sharing needed)
- **Opaque tokens** — fully stateless revocation via Redis, but requires a DB lookup on every request
- **Session cookies** — classic, but doesn't work well across multiple API services

## Decision

We will use **RS256 JWT access tokens (15 min) + refresh token rotation with family revocation**.

Design:

1. **Access token:** RS256-signed JWT, 15-minute TTL. Services verify with the public key — no DB lookup needed on each request
2. **Refresh token:** Stored as `httpOnly + SameSite=Strict + Secure` cookie. 7-day TTL. Rotated on every use
3. **Token family:** Each refresh token belongs to a family (UUID). If a refresh token is used after rotation, the entire family is revoked — this detects token theft
4. **Redis blacklist:** Revoked JTIs are stored in Redis with the token's remaining TTL. Access tokens are checked against this on each request
5. **RS256 asymmetric keys:** Auth service holds the private key. All other services hold the public key — secrets never leave the auth service

## Consequences

- ✅ Services can verify tokens without calling the auth service on every request
- ✅ Refresh token reuse detection catches stolen tokens and invalidates the entire session
- ✅ Refresh tokens are httpOnly — not accessible to JavaScript (XSS protection)
- ✅ Immediate revocation via Redis blacklist (access token) and database deletion (refresh token)
- ⚠️ Redis must be available for the blacklist check on every authenticated request — plan for Redis HA
- ⚠️ RS256 key rotation requires coordinated deployment across all services
- ⚠️ Access tokens cannot be immediately revoked without the Redis blacklist (mitigated by 15-min TTL)
