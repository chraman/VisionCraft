# ADR-0009: Safety Middleware as Pass-Through Slot in v1

**Status:** Accepted
**Date:** 2025-01-01

## Context

AI image generation services must consider content safety — preventing generation of harmful, NSFW, or illegal content. Options for v1:

1. **No safety check** — fastest to ship, but creates legal and reputational risk
2. **Block certain keywords** — naive, easily bypassed, false positives
3. **Third-party safety API** — adds latency and cost to every request
4. **Pass-through slot** — implement the interface now, fill in the implementation in Phase 3

## Decision

We will implement a **safety middleware slot** in v1 that is a deliberate pass-through.

The `safety_check()` function in `ai-service/middleware/safety.py` has a fixed interface:

```python
async def safety_check(
    prompt: str | None,
    image_bytes: bytes | None,
) -> SafetyResult:
    # v1: always returns passed=True
    return SafetyResult(passed=True, score=0.0, reason=None)
```

The caller does not change when safety is enabled. Only this function body changes. The function is activated via:

- `SAFETY_ENABLED=true` environment variable
- `ai.safety_check.enabled` feature flag

In Phase 3, we will implement the classifier body — likely using NudeNet or a Claude-powered safety check.

## Consequences

- ✅ The interface is fixed — all callers are written correctly from day one
- ✅ No latency cost in v1 (the function is a no-op)
- ✅ Activating safety in Phase 3 requires only implementing one function body
- ✅ Feature flag allows per-user or per-tier safety enforcement
- ⚠️ v1 has no content safety — terms of service and user agreement must clearly prohibit harmful use
- ⚠️ Legal risk: platform operator is responsible for generated content without safety checks
- ⚠️ Must implement before opening to public (Phase 3 gate)
