# ADR-0001: Use Turborepo over Nx

**Status:** Accepted
**Date:** 2025-01-01

## Context

The platform is a monorepo with multiple apps (web, admin), shared packages, Node.js services, and a Python AI service. We need a monorepo build system that:

- Caches build artifacts to speed up CI and local development
- Supports incremental builds (only rebuild what changed)
- Integrates natively with our chosen deployment platforms (Vercel, Railway)
- Has low configuration overhead

The two primary candidates were **Turborepo** and **Nx**.

## Decision

We will use **Turborepo** as the monorepo build system.

Key factors:

1. **Vercel-native:** Turborepo Remote Cache is free and built into Vercel — zero configuration needed for CI caching
2. **Simpler configuration:** `turbo.json` is significantly simpler than Nx's plugin/executor system
3. **No lock-in:** Turborepo only orchestrates task running — it doesn't generate code or impose file structure
4. **pnpm workspace compatibility:** Works seamlessly with pnpm workspaces, our chosen package manager

## Consequences

- ✅ Build caching works out of the box on Vercel and GitHub Actions (via `TURBO_TOKEN`)
- ✅ Minimal configuration — `turbo.json` is small and readable
- ✅ Any npm/pnpm command can be orchestrated by Turborepo
- ⚠️ Fewer built-in generators than Nx — we scaffold manually
- ⚠️ No built-in dependency graph visualisation (Nx has this)
