# ADR-0006: Prisma with Soft Deletes

**Status:** Accepted
**Date:** 2025-01-01

## Context

User-generated content (images, collections) must never be permanently deleted. Requirements:

- Compliance: users must be able to export or restore deleted content
- Safety: prevent accidental permanent data loss
- Audit: maintain a complete record of all data mutations

Options considered:

- **Hard deletes** — simple, but data is gone forever
- **Soft deletes with Prisma** — set `deletedAt`, filter on queries
- **Event sourcing** — append-only log, complex to query
- **Archive table** — move records to separate table on delete

## Decision

We will use **soft deletes throughout** — all user-owned records have a `deletedAt` column. Deleting a record sets `deletedAt = now()` via a repository method.

Rules:

1. Never call `.delete()` on user-owned records — always use `updateOne({ deletedAt: new Date() })`
2. All list queries automatically filter `WHERE deletedAt IS NULL`
3. Repository classes enforce this — no raw Prisma calls in route handlers
4. The `version` field enables optimistic locking for concurrent updates

## Consequences

- ✅ All user content is recoverable by setting `deletedAt = NULL`
- ✅ Complete history of all records
- ✅ Compliance with data regulations requiring right-to-erasure (we can actually delete after 30 days via a cleanup job)
- ⚠️ All queries must include `WHERE deletedAt IS NULL` — enforced via repository layer
- ⚠️ Database grows over time — a periodic cleanup job for old soft-deleted records is needed (Phase 2)
- ⚠️ Unique constraints must account for soft deletes (e.g., `email` uniqueness only among non-deleted users)
