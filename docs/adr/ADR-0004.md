# ADR-0004: LaunchDarkly (prod) + Unleash (dev) for Feature Flags

**Status:** Accepted
**Date:** 2025-01-01

## Context

Feature flags are a first-class architectural concern (CLAUDE.md §5). Every user-facing feature is behind a flag. We need:

- Real-time flag updates (< 1s) in production
- Targeting rules (user tier, userId, % rollout)
- A/B test support
- Zero network dependency in tests (use static JSON)
- Low cost for early-stage development

Options considered:

- **LaunchDarkly** — industry standard, expensive at scale, < 1s streaming updates
- **Unleash** — self-hostable, free, same feature set
- **GrowthBook** — open source, focused on experimentation
- **Homegrown** — Redis-backed flag store

## Decision

- **Production/Staging:** LaunchDarkly (SDK key from Railway secrets)
- **Development:** Unleash self-hosted in Docker Compose
- **CI/Tests:** Static `flags.test.json` — no network, fully deterministic
- **Fallback:** `flags.default.json` bundled in `packages/feature-flags` — used on deploy before provider connects

Key factors:

1. LaunchDarkly's streaming SDK delivers < 1s flag changes — critical for canary deployments
2. Unleash in Docker Compose costs nothing in development
3. The `FlagClient` abstraction in `packages/feature-flags` means provider switching is a one-line change

## Consequences

- ✅ < 1s flag propagation in production
- ✅ Free local development with full feature parity
- ✅ Deterministic tests with no network dependency
- ✅ Provider can be swapped without changing calling code
- ⚠️ LaunchDarkly has a cost at scale (first 1000 MAU free)
- ⚠️ Two flag providers means keeping LaunchDarkly and Unleash in sync during development
