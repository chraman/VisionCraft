# ADR-0008: Railway + Vercel over Kubernetes

**Status:** Accepted
**Date:** 2025-01-01

## Context

The platform needs to deploy and operate multiple services (6 Node.js services, 1 Python service, 1 worker) with managed databases and zero-ops overhead for a small team. Options considered:

- **Railway + Vercel** — PaaS, managed everything, fast setup
- **AWS ECS / Fargate** — managed containers, more control, more ops
- **Kubernetes (EKS/GKE)** — maximum control, significant ops burden
- **Render** — similar to Railway, fewer features

## Decision

We will use **Vercel** for the React frontend and **Railway** for all backend services.

Key factors:

1. **Zero-ops:** Railway manages PostgreSQL, Redis, service discovery, TLS, auto-scaling, and rolling deployments — no infrastructure team needed
2. **Private networking:** Railway services communicate via `.railway.internal` hostnames — no public internet exposure for internal calls, no extra VPC configuration
3. **Git-driven deploys:** Push to main → automatic staging deploy → manual promote to production. Works out of the box
4. **Managed databases:** Railway PostgreSQL and Redis include daily backups, point-in-time restore, and automatic failover
5. **Speed:** Total setup time is hours, not weeks. We're optimising for iteration speed at this stage
6. **Vercel + Turborepo:** Native remote cache integration, automatic preview deployments per PR

## Consequences

- ✅ Zero infrastructure ops — focus entirely on product
- ✅ Private networking between services without VPN/VPC configuration
- ✅ Automatic preview URLs for every PR
- ✅ Managed DB backups and restore
- ⚠️ Less control than Kubernetes (no custom networking, no sidecar containers)
- ⚠️ Railway pricing scales with usage — revisit at 100k+ users
- ⚠️ No GPU support on Railway — Phase 4 GPU inference will require a dedicated GPU service (RunPod, Lambda Labs)
