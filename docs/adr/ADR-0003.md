# ADR-0003: FastAPI for AI Service

**Status:** Accepted
**Date:** 2025-01-01

## Context

The AI generation service needs to orchestrate multiple AI providers (Stability AI, OpenAI, HuggingFace). Options considered:

- **FastAPI** (Python)
- **Express** (Node.js) — consistent with other services
- **gRPC service** (Python or Go)

## Decision

We will use **FastAPI with Python 3.11** for the AI service, even though all other services are Node.js.

Key factors:

1. **Python ML ecosystem:** Stability AI SDK, diffusers, Pillow, and HuggingFace libraries are all Python-first. Node.js bindings are often incomplete or unmaintained
2. **Async support:** FastAPI is async-native with uvicorn — handles concurrent AI API calls efficiently
3. **Future GPU support:** When we self-host models (Phase 4), Python is the only viable choice — PyTorch, CUDA bindings, diffusers all require Python
4. **Auto-generated docs:** FastAPI generates OpenAPI docs automatically — keeping API contracts in sync with Node.js consumers
5. **Performance:** For I/O-bound AI API calls, Python async is competitive with Node.js

## Consequences

- ✅ Access to full Python ML ecosystem
- ✅ Smooth path to self-hosted GPU inference in Phase 4
- ✅ Automatic OpenAPI spec generation
- ⚠️ Two language ecosystems to maintain (Node.js + Python) — additional developer context switching
- ⚠️ Shared types must be manually kept in sync (or generated from OpenAPI spec)
- ⚠️ Python cold-start time is longer than Node.js on Railway
